# oca-docs-reader

> OCA Harness 文档阅读指南。告诉 LLM 在什么场景下该读哪个 OCA-*.md 文档。

## 何时使用

当你（LLM）需要了解项目结构、功能、接口或规则时，根据任务类型选择正确的文档阅读。避免读错文档浪费时间，也避免不读文档导致违反项目约定。

## OCA Harness 文档体系

OpenCrossAgent 使用两级 harness 文档体系：

| 层级 | 位置 | 职责 |
|------|------|------|
| **项目级** | `./OCA-*.md` | 跨模块通用的全局信息 |
| **模块级** | `gateway/OCA-*.md`、`clients/cli/OCA-*.md`、`installer/OCA-*.md` | 该模块特有的详细信息 |

### 四类文档

| 文档 | 定位 | 读它的场景 |
|------|------|-----------|
| **OCA-ARCH** | 架构设计 | 你需要知道目录结构、模块划分、依赖方向（允许的方向）、技术栈选型、某个功能在哪个目录时 |
| **OCA-FEATURE** | 功能清单 | 你需要知道项目/模块提供了哪些功能、构建命令/产物、运行时架构、配置模板时 |
| **OCA-API** | 接口契约 | 你需要调用其他模块的接口、复用已有类型签名、了解导出 API 时。**接口签名必须与实现计划完全一致。** |
| **OCA-RULE** | 项目规则 | 你在编写/修改代码前，需要知道哪些约束必须遵守、哪些行为被禁止（包括禁止的 import 方向）时 |

### ARCH 与 RULE 的分工

避免在两个文档间来回查找依赖方向信息：

| 你需要的信息 | 读哪个文档 |
|-------------|-----------|
| 允许的依赖方向（A 可以依赖 B） | OCA-ARCH §依赖方向 |
| 禁止的 import 方向（A 不能 import B） | OCA-RULE §依赖方向红线 |
| 层内依赖规则（每层只能依赖什么） | OCA-ARCH §依赖方向（RULE 不重复，仅交叉引用） |

### ARCH 与 FEATURE 的分工

| 你需要的信息 | 读哪个文档 |
|-------------|-----------|
| 目录结构和模块划分 | OCA-ARCH |
| 技术栈选型（语言/运行时/包管理/构建工具） | OCA-ARCH |
| 包命名和 bin 命令 | OCA-FEATURE |
| 架构设计约束（如"不拆子 package"） | OCA-ARCH §架构设计约束 |
| 功能清单和实现状态 | OCA-FEATURE |
| 构建命令和产物 | OCA-FEATURE |
| 配置模板结构 | OCA-FEATURE |

## 阅读决策树

```
你的任务是什么？
│
├── "我要修改某处代码"
│   → 先读 OCA-RULE（了解约束，特别是禁止的 import 方向）
│   → 再读目标模块的 OCA-ARCH（定位文件，了解允许的依赖方向）
│   → 如需调用其他模块，读 OCA-API（查接口签名）
│
├── "我要新增一个功能/模块"
│   → 先读根 OCA-ARCH（了解架构边界和设计约束）
│   → 读根 OCA-RULE（了解依赖方向红线和明确不做）
│   → 读根 OCA-FEATURE（了解已有功能，避免重复）
│   → 读目标模块的 OCA-API（了解已有接口，确保复用）
│
├── "我要理解项目做什么"
│   → 读根 OCA-FEATURE（功能清单 + 技术栈）
│   → 读根 OCA-ARCH（架构总览）
│
├── "我要构建/部署项目"
│   → 读目标模块的 OCA-FEATURE（构建命令、产物、配置模板）
│
├── "我要调用某模块的接口"
│   → 读目标模块的 OCA-API（接口签名、类型定义）
│   → 如需理解接口上下文，读该模块的 OCA-ARCH
│   → 注意：接口签名是契约，必须与实现计划一致
│
└── "我犯了错，要记录教训"
    → 读 OCA-RULE 的 §故障教训记录（确认格式）
    → 追加 SIGN 记录（但注意：OCA-*.md 文件 AI 不得自动修改，需告知用户手动追加）
```

## 阅读优先级

当多个文档都相关时，按以下优先级阅读：

1. **OCA-RULE** — 最高优先级。先了解约束，避免一开始就犯错。
2. **OCA-ARCH** — 了解结构，定位目标文件。
3. **OCA-API** — 查阅接口，确保调用正确。
4. **OCA-FEATURE** — 了解功能上下文，按需阅读。

## 模块级 vs 项目级

- **先读项目级**：了解全局架构和通用规则。
- **再读模块级**：了解目标模块的具体细节。模块级文档位置：`gateway/OCA-*.md`、`clients/cli/OCA-*.md`、`installer/OCA-*.md`。
- 模块级文档是项目级的细化，不重复项目级内容。如果项目级已回答你的问题，不需要再读模块级。
- 随着模块实现推进，项目级内容会逐步下沉到模块级。如果项目级文档中找不到某模块的细节，检查该模块目录下是否有模块级文档。

## 注意事项

- OCA-*.md 文件是 **AI 不得自动修改** 的（见 OCA-RULE §MUST NOT #3）。
- 如果发现文档内容与代码不一致，告知用户，不要自行修改文档。
- 文档中的接口签名是契约，代码实现必须与之匹配。如需变更接口，先告知用户。
- 如果发现接口签名与实现计划（brainstorms/plan-*.md）不一致，告知用户需要同步。
