# oca-docs-writer

> OCA Harness 文档编写指南。告诉 LLM 每类 OCA-*.md 文档应该写什么、不应该写什么。

## 何时使用

当你（LLM）需要创建或更新 OCA-*.md harness 文档时，使用本指南确保内容归属正确，避免文档间重复或内容错位。

## 四类文档的职责边界

### OCA-ARCH — 架构设计

**写什么：**
- 目录结构和模块划分
- 模块间的依赖方向（用图或列表表示）
- 每个目录的 3W 说明（Why 存在 / What 包含 / When 修改）
- 架构原则（如"目录即架构"）
- 架构设计约束（如"四层通过目录划分不拆子 package"）
- 技术栈选型表（语言、运行时、包管理、构建工具、模块系统）— 技术栈是架构决策的基础

**不写什么：**
- ❌ 接口签名和类型定义（→ OCA-API）
- ❌ 构建命令和构建产物（→ OCA-FEATURE）
- ❌ 约束和禁止规则（→ OCA-RULE）
- ❌ 功能列表和状态（→ OCA-FEATURE）

**边界判定 — "架构上下文" vs "功能描述"：**
- ✅ "gateway 内部四层通过目录划分" — 架构决策，写在这里
- ✅ "依赖方向：channel → core → provider ← backend" — 架构约束，写在这里
- ✅ "技术栈：TypeScript + Node.js >= 22 + pnpm + ESM" — 架构决策，写在这里
- ❌ "3 个顶层 package" — 既是架构也是功能，已在 FEATURE 的包命名表中，不重复
- ❌ "新增 Channel 只需在 channel/ 下新增子目录" — 这是扩展能力描述，属于 FEATURE 的功能清单

**项目级 vs 模块级：**
- 项目级：四层架构总览 + 依赖方向图 + 架构原则 + 技术栈
- 模块级：模块内部的 3W 拆分细则 + 内部依赖流

---

### OCA-FEATURE — 功能清单

**写什么：**
- 包命名与 bin 命令
- 功能清单表（功能名 / 状态 / 说明）
- 构建概览（项目级：`pnpm install` / `pnpm -r build` 等基本命令）
- 构建配置细节（模块级：tsdown entries、onSuccess hook、externalize 等）
- 构建产物结构（各 package dist/ 下的文件清单）
- 运行时架构（进程关系图）
- 版本管理策略
- 配置模板结构说明

**不写什么：**
- ❌ 目录结构和依赖方向（→ OCA-ARCH）
- ❌ 技术栈选型（→ OCA-ARCH，技术栈是架构决策）
- ❌ 接口签名和类型定义（→ OCA-API）
- ❌ 约束和禁止规则（→ OCA-RULE）
- ❌ 编码规范（ESM 规则、测试规则等 → OCA-RULE）

**项目级 vs 模块级：**
- 项目级：包命名 + 全项目功能清单 + 构建概览 + 运行时架构 + 版本管理 + 配置模板结构
- 模块级：模块特有的功能列表 + 构建配置细节 + 构建产物 + dev 工作流

---

### OCA-API — 接口契约

**写什么：**
- 接口/类型/函数的签名定义（TypeScript 代码块）
- 每个导出的用途说明
- 模块间的调用关系
- 新增扩展时的契约要求（如"新增 Backend 需实现 IAgentProvider"）

**不写什么：**
- ❌ 目录结构和依赖方向（→ OCA-ARCH）
- ❌ 功能描述和构建配置（→ OCA-FEATURE）
- ❌ 约束和禁止规则（→ OCA-RULE）
- ❌ 实现细节（函数内部逻辑）

**同步规则：**
- OCA-API 中的类型定义必须与实现计划（brainstorms/plan-*.md）中定义的类型**完全一致**
- 如果计划中定义了 14 种事件类型，API 文档中必须列出全部 14 种，不得遗漏
- 新增类型时，先更新 API 文档，再更新代码

**项目级 vs 模块级：**
- 项目级：跨模块共享的类型（如 `AgentEvent`、`Session`、`ModelInfo`）
- 模块级：该模块各文件的导出接口签名 + 类型定义

---

### OCA-RULE — 项目规则

**写什么：**
- 宪法规则（MUST / MUST NOT / SHOULD）
- 依赖方向红线（禁止的 import 方向表）
- 架构红线（依赖单向、接口隔离、单一职责）
- 文件边界约束（禁止修改的文件列表）
- 编码约束（ESM 规则、测试规则、资产分离）
- 明确不做（决策排除项）
- 故障教训记录（SIGN 格式）

**不写什么：**
- ❌ 目录结构和依赖方向图（→ OCA-ARCH，但"禁止的方向"写在这里）
- ❌ 功能列表和构建配置（→ OCA-FEATURE）
- ❌ 接口签名（→ OCA-API）
- ❌ 实现逻辑描述

**与 ARCH 的边界 — "允许" vs "禁止"：**
- ARCH 写"允许的依赖方向"（正向描述 + 图）
- RULE 写"禁止的 import 方向"（反向描述 + 表）
- **不要在 RULE 中重复 ARCH 的允许列表**，用交叉引用链接：`允许的依赖方向详见 OCA-ARCH.md`

**项目级 vs 模块级：**
- 项目级：宪法 + 依赖方向红线 + 全局编码约束 + 文件边界 + 明确不做
- 模块级：模块内部特有的规则（如"MCP Server 只允许只读工具"）

## 编写原则

1. **不重复** — 如果某内容已在另一文档中，不要复制。用交叉引用链接。
2. **跨文档去重** — 编写时检查同一内容是否已在其他 OCA-*.md 中出现。特别关注 ARCH 和 RULE 之间的依赖方向描述、ARCH 和 FEATURE 之间的包结构描述。
3. **项目级精简** — 项目级文档只保留跨模块通用的内容，模块细节下沉到模块级。
4. **模块级细化** — 模块级文档是该模块的"说明书"，包含该模块所有特有信息。
5. **接口即契约** — OCA-API 中的签名是代码必须遵守的契约，必须与实现计划完全一致，变更需同步告知用户。
6. **规则即护栏** — OCA-RULE 中的约束是硬性的，不是建议。
7. **功能即能力** — OCA-FEATURE 描述"能做什么"，不是"怎么约束"。

## 纠错迁移规则

当发现已有 OCA-*.md 文档中存在**内容归属错误**（内容放在了错误的文档中）时：

1. **迁移内容** — 将内容从错误的文档中删除，添加到正确的文档中
2. **添加交叉引用** — 在原文档中添加指向新位置的链接（如"技术栈见 [OCA-ARCH.md](./OCA-ARCH.md) §技术栈"）
3. **更新文档头部描述** — 如果文档头部描述中提到了被移走的内容，同步更新
4. **告知用户** — 列出所有迁移的内容和涉及的文档

**常见归属错误及修正：**

| 错误 | 错误位置 | 正确位置 | 原因 |
|------|---------|---------|------|
| 技术栈表 | OCA-FEATURE | OCA-ARCH | 技术栈是架构决策（选什么技术栈决定了目录结构和模块系统），不是功能 |
| 层内依赖规则 | OCA-RULE | OCA-ARCH | "允许的依赖"属于架构描述，RULE 只写"禁止的" |
| 构建命令 | OCA-ARCH | OCA-FEATURE | 构建命令是操作流程，不是架构设计 |
| ESM 编码规则 | OCA-FEATURE | OCA-RULE | 编码规则是约束，不是功能 |
| 功能状态 | OCA-ARCH | OCA-FEATURE | 功能列表和实现状态是能力描述 |

## 编写后检查清单

完成文档编写/更新后，逐项检查：

- [ ] **类型完整性** — 如果是 OCA-API，所有计划中定义的类型是否都已列出？
- [ ] **跨文档去重** — 同一内容是否在多个文档中重复出现？如果是，保留最合适的一个，其他改为交叉引用。
- [ ] **归属正确** — 每段内容是否放在了正确的文档中？（用内容归属速查表核对）
- [ ] **纠错迁移** — 是否有内容在错误的文档中？如果有，迁移到正确位置并添加交叉引用。
- [ ] **交叉引用** — 被移除的内容是否有对应文档接收？是否添加了链接？
- [ ] **项目级 vs 模块级** — 项目级文档是否还包含应下沉到模块级的内容？

## 重要约束

- **AI 不得自动修改 OCA-*.md 文件**（OCA-RULE §MUST NOT #3）。
- 如果代码变更导致文档需要更新，告知用户需要手动更新哪些文档的哪些部分。
- 新建模块时，告知用户需要创建该模块的 4 个 OCA-*.md 文件。
